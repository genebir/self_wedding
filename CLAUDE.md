## 0. 프로젝트 코드명: 맑음 (Malgeum) — *작업용 가칭, 변경 가능*

"맑음"은 가격이 깜깜이인 웨딩 시장을 *맑게* 한다는 뜻이자, 결혼 준비가 *맑고 즐거운* 시간이길 바라는 마음이다.

---

## 1. 우리가 진짜 만드는 것

**맑음은 셀프 웨딩 준비 OS다. 플래너가 하던 일을 도구로 대체하고, 사용자들의 실지출이 쌓여 시장의 가격을 맑게 한다.**

한국에서 셀프 준비가 어려운 이유는 하나가 아니라 세 겹이다:

1. **가격 비대칭** — "상담 후 견적", 부르는 게 값. 남들 얼마 내는지 모름.
2. **구조적 차단** — 스드메 상당수가 플래너 경유 가격이 더 싸거나 직거래를 안 받음. 개인은 협상력이 없음.
3. **조율 노동 + 숨은 비용 지뢰밭** — 헬퍼비·원본비·피팅비·얼리스타트비 같은 옵션비, 일정 조율, 누락 체크를 플래너가 대신 해주는 게 실질 가치.

가격 투명화(1)만 공격하는 제품은 반쪽이다. 맑음은 **1+2+3 전부를 도구로 상대한다.**
셀프 준비가 본체고, 가격 투명화는 셀프 준비를 가능하게 하는 **수단이자 해자**다.

- **목적:** 플래너 없이도 준비 과정이 투명하고 즐겁게.
- **목적이 아닌 것:** "결혼을 싸게" (집값 등은 우리 밖의 문제). 우리는 *정보 비대칭과 조율 노동에서 오는 공포와 피로*만 정직하게 손댄다.

## 2. 리트머스 테스트 (모든 결정의 기준선)

새 기능·코드·카피·수익모델은 **반드시** 세 질문을 통과해야 한다. 하나라도 실패하면 만들지 않는다.

1. **이것은 셀프 준비를 실제로 쉽게 만드는가?** (사용자 1명일 때도 가치가 있는가)
2. **이것은 소비자 편인가?** (업체·중개가 아니라)
3. **이것은 데이터의 정직함을 지키는가?** (표본·출처·조건을 숨기지 않는가)

여기서 따라 나오는 절대 원칙:

- **🚫 수익모델은 절대 업체 광고·중개·리드 판매가 될 수 없다.** 그 순간 "소비자 편"이라는 명분이 자기 비즈니스와 충돌하고, 우리는 기존 웨딩 서비스와 똑같아진다. 수익은 **소비자 쪽 부가가치**(프리미엄 도구, 견적 분석, 예산 리포트 등)에서만 나온다.
- **🚫 다크패턴 금지.** 가짜 urgency, 숨은 비용, 비교 방해, 강제 가입 후 정보 노출 — 전부 우리가 싸우는 대상이다. 만들지 않는다.
- **✅ 임계 미달 데이터는 공개하지 않는다.** 표본이 얇은 분포를 그럴듯하게 보여주는 것은 거짓말이다. (→ 5.3)

## 3. 제품 구조: 툴-퍼스트, 데이터는 배기가스

v1의 급소는 콜드스타트였다: 데이터-퍼스트 플랫폼은 표본이 임계점을 넘기 전까지 빈 약속이다.
v2는 순서를 뒤집는다. **도구가 먼저 가치를 주고, 데이터는 도구 사용의 배기가스(exhaust)로 쌓인다.**

### Phase 0 — 도그푸딩 (지금)
메인테이너 본인의 결혼 준비(2027년 3월, 수도권)가 첫 데이터이자 첫 명세다.
플래너 미팅·견적·실지출·짜증 포인트를 전부 기록한다. **플래너가 해주는 일 목록 = OS가 대체할 기능 명세.**

### Phase 1 — 싱글플레이어 OS (콜드스타트 없음)
사용자 1명일 때도 완결된 가치. 네트워크 효과에 의존하는 기능은 이 단계에 넣지 않는다.

- **예산 트래커 (척추).** 항목별 예상/실지출, `scope`와 `attributes`가 입력 폼에 내장. 사용자는 자기 돈을 관리하고, 그 행위가 곧 구조화된 실거래 데이터를 만든다.
- **체크리스트/타임라인.** 예식일 역산 캘린더. 예약 데드라인, 누락 경고. 플래너의 "조율 노동" 대체 1호.
- **숨은비용 사전.** 헬퍼비·원본비·피팅비·얼리스타트비 등 옵션비 지뢰의 정적 콘텐츠 사전. 네트워크 효과 0으로 신뢰 브랜드와 SEO 유입을 만드는 지렛대. 모든 항목은 예산 트래커의 `scope` 체크리스트와 연동된다.

### Phase 2 — 견적 디코더 (성장 웨지 + 고신뢰 수집기)
견적서 캡처 업로드 → OCR+LLM 구조화 → 숨은비용 사전 기반으로 지뢰 플래그 + 포함/불포함 명세화.
- 사용자에겐 "이 견적 어때?"에 대한 즉답 (공유하고 싶어지는 순간 = 바이럴 포인트)
- 우리에겐 **인증 견적**이라는 최고 신뢰 등급 데이터 (→ 5.4)

### Phase 3 — 분포 공개 (임계 잠금해제)
카테고리×지역 단위로 표본이 임계(n≥10)를 넘는 곳부터 실거래 분포를 연다.
"○○명이 실제로 낸 금액 기준 · 중앙값 ○○만원" — 이 문장을 쓸 수 없으면 그 화면은 만들지 않는다.
분포가 열리면 셀프 준비의 마지막 퍼즐(협상 기준선)이 맞춰진다: *"이 카테고리 중앙값이 X인데 견적이 X+40%네요"*가 개인의 협상력이 된다. 구조적 차단(1장 2번)을 정면 돌파하는 무기.

**단계 승격 규칙:** 다음 Phase의 기능은 이전 Phase가 도그푸딩 검증을 통과하기 전에 만들지 않는다.

## 4. 데이터 기여: 강요가 아니라 동의

v1의 give-to-get("넣어야 본다")을 **동의 기반 교환**으로 다듬는다.

- 예산 트래커 데이터는 **기본 비공개, 사용자 소유**다. 트래커를 쓰기 위해 기여를 강요받지 않는다.
- 분포 열람은 교환이다: *"내 지출을 익명 통계에 보태면, 남들의 분포를 볼 수 있어요"* — 명시적 opt-in, 언제든 철회 가능, 철회 시 집계에서 제거.
- 기여 시 무엇이 어떻게 익명화되는지 한 화면으로 보여준다. 숨기는 게 없어야 신뢰가 쌓인다.

이 구조가 신뢰와 확장을 동시에 잡는다: 강요된 데이터는 쓰레기가 섞이고, 동의된 데이터는 깨끗하다.

## 5. 데이터 철학 (이 제품의 심장)

### 5.1 가격은 "정가"가 아니라 "실거래 분포"다
업체 공시가는 존재하지 않거나 거짓이다. 단일값(min/max) ❌ → **표본 기반**(중앙값·사분위·표본수·신뢰 등급) ✅

### 5.2 데이터의 원천 우선순위 (v2에서 역전됨)
1. **도구 배기가스 (주력).** 예산 트래커의 실지출 — 사용자가 *자기를 위해* 넣은 숫자라 품질의 출발점이 다르다.
2. **견적 디코더 (고신뢰).** 원본 증빙이 있는 인증 견적.
3. **커뮤니티 후기 마이닝 (보강).** 공개 후기에서 *사실(항목·지역·금액·시기)만* 추출. 본문 저장·재공개 금지.

### 5.3 임계 규칙 — 얇은 표본은 침묵한다
- 카테고리×지역 표본 **n < 10: 분포 비공개** ("아직 표본이 부족해요. 기여하면 빨라져요"로 정직하게)
- n ≥ 10: 공개하되 표본수·기간·신뢰 등급 상시 표기
- 모든 분포 화면에 "이 숫자는 어떻게 만들어졌나" 링크 (산출 방법 공개)

### 5.4 신뢰 등급 — 모든 데이터 포인트는 출처 등급을 가진다
| 등급 | 원천 | 가중치 |
|------|------|--------|
| A (인증) | 견적 디코더 원본 증빙 | 최고 |
| B (트래커) | 예산 트래커 실지출 | 높음 |
| C (자가 보고) | 수기 입력 단건 | 보통 |
| D (마이닝) | 공개 후기 추출 | 참고 |
분포 계산은 등급 가중, 화면에는 등급 구성 표기.

### 5.5 오염 방어 — 업체는 이 데이터를 공격할 동기가 있다
자기 가격은 낮게, 경쟁사는 높게 보이게 할 유인이 존재한다. 방어는 기능이 아니라 전제다:
- 이상치 탐지 (카테고리별 분포 대비 자동 플래그, 사람 검토 큐)
- 계정당 기여 rate limit, 단기간 동일 업체 집중 기여 탐지
- A등급(증빙) 데이터가 앵커 — 자가 보고가 인증 견적 분포와 크게 어긋나면 신뢰 하향
- 오염 시도가 확인된 업체는 내부 기록 (공개 비방은 하지 않는다 — 우리는 데이터로만 말한다)

### 5.6 비용 데이터 신뢰의 단일 급소: `scope`(포함 범위)
"스냅 100만원"이 촬영만인지 앨범·보정 포함인지 모르면 비교 불가다.
- **모든 지출·견적 레코드는 `scope`를 1급 필드로 반드시 받는다.** 숨은비용 사전이 곧 `scope` 체크리스트의 원천.
- 항목별 가격 변수(스냅의 시간·렌즈, 드레스의 벌수·브랜드급 등)는 `attributes JSONB`로 흡수한다.

### 5.7 업체는 선등록하지 않는다 — 데이터에서 자라난다
크롤링 선등록 ❌. 사용자가 업체명을 적으면 그 순간 `vendor` upsert. 등록된 업체는 전부 실데이터 1건 이상.

### 5.8 크롤링은 회색지대다 — 선을 지킨다
- ✅ robots.txt·약관 존중, 공개 페이지만, rate limit 준수
- ✅ 본문 복제·전시 ❌ → 사실 데이터만 추출
- 🚫 폐쇄형 카페 로그인/접근통제 우회 코드는 절대 만들지 않는다
- 🚫 작성자 개인정보(연락처 등) 절대 수집 금지

### 5.9 익명화는 k-익명성 기준
지역×시기×업체×금액 조합으로 개인이 특정될 수 있다. 공개 집계는 셀 단위 k≥5 미만이면 상위 버킷으로 병합. "익명"이라고 말했으면 역추적이 불가능해야 한다.

## 6. 기술 스택과 데이터 모델

- **PostgreSQL 단일** (MVP). Neo4j는 보류 — 관계 질의가 실제 병목이 되기 전까지 도입하지 않는다.
- **FastAPI + Next.js.** 백엔드는 얇게, 데이터 모델에 투자한다.
- OCR+LLM 파이프라인은 Phase 2 전까지 만들지 않는다 (승격 규칙).

핵심 테이블 (개념):

```
taxonomy        카테고리 트리 (스드메/예식장/스냅/한복/... versioned)
expense         예산 트래커 항목 — 사용자 소유, 기본 비공개
                (category, planned, actual, scope[], attributes jsonb, vendor_id?, paid_at)
contribution    expense/quote에서 opt-in으로 파생된 익명 스냅샷 (trust_grade, region, period)
quote           견적 디코더 결과 (원본 증빙 해시, 구조화 라인아이템, flags[])
vendor          upsert-only. 실데이터에서만 생성
fee_glossary    숨은비용 사전 — scope 체크리스트의 원천
```

원칙: `expense`(사적 도구)와 `contribution`(공적 통계)을 물리적으로 분리한다. 도구 데이터가 사용자 동의 없이 통계로 새는 경로가 스키마 수준에서 존재하면 안 된다.

## 7. 작업 방식 (반드시 상시 적용)

### 7.1 도그푸딩 — 1호 사용자는 우리 자신이다
- 새 기능마다 묻는다: **"내가 지금 내 결혼 준비에 이걸 실제로 쓸까? 안 쓴다면 왜?"**
- 가짜 데이터로 데모를 채우지 않는다. 본인이 실제 겪은 가격·항목·짜증 포인트가 첫 데이터·첫 기능이다.
- 본인이 안 쓸 기능은 만들지 않는다.

### 7.2 페르소나 테스트 — 모든 기능은 페르소나 세션으로 검증한다
기능을 "완성"이라 부르기 전에 `PERSONAS.md`의 각 페르소나로 첫 세션을 끝까지 걸어본다. 특히 **적대적 페르소나(오염 시도 업체, 프라이버시 민감 사용자)를 통과 못 하면 완성이 아니다.**

### 7.3 신뢰는 카피에서도 지킨다
과장 금지. "최저가 보장" 같은 말은 우리 어휘에 없다. 우리가 아는 것과 모르는 것을 구분해서 말한다. 표본이 부족하면 부족하다고 쓴다.
