## 0. 프로젝트 코드명: 맑음 (Malgeum) — *작업용 가칭, 변경 가능*

"맑음"은 가격이 깜깜이인 웨딩 시장을 *맑게* 한다는 뜻이자, 결혼 준비가 *맑고 즐거운* 시간이길 바라는 마음이다.

---

## 1. 우리가 진짜 만드는 것

**맑음은 셀프 웨딩 준비 커뮤니티다. 서로의 준비 기록이 서로의 기준선이 되고, 그 기록에서 시장의 가격이 맑아진다.**

한국에서 셀프 준비가 어려운 세 겹의 문제는 그대로다:

1. **가격 비대칭** — "상담 후 견적", 부르는 게 값. 남들 얼마 내는지 모름.
2. **구조적 차단** — 스드메 상당수가 플래너 경유 가격이 더 싸거나 직거래를 안 받음. 개인은 협상력이 없음.
3. **조율 노동 + 숨은 비용 지뢰밭** — 헬퍼비·원본비·피팅비 같은 옵션비, 일정 조율, 누락 체크.

v2는 이걸 "혼자서도 완결된 도구"로 풀려 했다. 하지만 이 문제들의 본질은 **혼자라는 것 자체**다.
플래너가 가진 힘은 도구가 아니라 *수백 커플의 사례를 알고 있다는 것*이다. 맑음은 그 사례 축적을
커뮤니티로 재현한다: **"나 이렇게 준비했고, 이렇게 냈어요"가 오가는 공간.**

- **코어 루프:** 내 준비 과정을 기록·공유한다 → 다른 커플의 기록에서 배운다 → 내 기록이 누군가의 기준선이 된다.
- **목적:** 플래너 없이도 준비 과정이 투명하고 즐겁게 — 혼자가 아니라 서로 덕분에.
- **목적이 아닌 것:** "결혼을 싸게" (집값 등은 우리 밖의 문제). 우리는 *정보 비대칭과 조율 노동에서 오는 공포와 피로*만 정직하게 손댄다.

## 2. 리트머스 테스트 (모든 결정의 기준선)

새 기능·코드·카피·수익모델은 **반드시** 세 질문을 통과해야 한다. 하나라도 실패하면 만들지 않는다.

1. **이것은 서로의 준비 기록을 서로에게 더 유용하게 만드는가?** (공유할 이유, 읽을 이유를 늘리는가)
2. **이것은 소비자 편인가?** (업체·중개가 아니라)
3. **이것은 데이터의 정직함을 지키는가?** (표본·출처·조건을 숨기지 않는가)

여기서 따라 나오는 절대 원칙:

- **🚫 수익모델은 절대 업체 광고·중개·리드 판매가 될 수 없다.** 커뮤니티에서 이 선이 무너지면 "업체 바이럴이 섞인 카페"와 똑같아진다 — 우리가 대체하려는 바로 그것. 수익은 **소비자 쪽 부가가치**(프리미엄 도구, 견적 분석, 예산 리포트 등)에서만 나온다.
- **🚫 다크패턴 금지.** 가짜 urgency, 숨은 비용, 비교 방해, 강제 가입 후 정보 노출 — 전부 우리가 싸우는 대상이다. **피드와 사전은 가입 없이 읽을 수 있다.** 가입은 쓰기 위한 것이다.
- **✅ 임계 미달 데이터는 공개하지 않는다.** 표본이 얇은 분포를 그럴듯하게 보여주는 것은 거짓말이다. (→ 5.3)
- **🚫 업체(광고성) 계정의 후기 위장 콘텐츠는 커뮤니티의 1급 오염이다.** 방어는 기능이 아니라 전제. (→ 5.5)

## 3. 제품 구조: 피드가 척추, 도구는 재료

### 콜드스타트에 대한 정직한 태도
커뮤니티는 "빈 방" 문제가 있다. v2가 그걸 피해 설계됐지만, 그 대가로 공유의 의미를 잃었다.
v3는 빈 방을 감수하고 정면으로 푼다:
① 메인테이너 커플의 도그푸딩 기록이 첫 콘텐츠다 (2027년 3월 예식, 수도권 — 실제 기록만, 가짜 콘텐츠 절대 금지).
② 숨은비용 사전은 SEO 유입구로 유지한다 (검색으로 들어와 피드를 발견하는 동선).
③ 초기엔 소수 초대 기반으로 밀도를 지킨다. 빈 피드를 넓게 여는 것보다 좁고 진짜인 피드가 낫다.

### Phase A — 커뮤니티 코어 (지금)
- **계정.** 닉네임 기반. 실명·연락처를 요구하지 않는다 — 지출 공유는 민감하다.
- **준비 기록 피드.** 포스트 = 글 + (선택) **지출 카드**. "스드메 계약했어요, 이렇게 냈어요"가 대표 포스트.
- **지출 카드.** 카테고리·금액·scope(포함 범위)·업체·지역·시기(월 단위)가 구조화된 카드. 피드에 흐르는 실거래 데이터의 단위.
- **댓글.** 질문과 답이 오가는 곳. 여기서 숨은비용 정보가 자연 발생한다.
- **개인 도구(트래커·체크리스트)는 유지하되 재료로.** 트래커에 기록한 지출을 한 번의 탭으로 카드로 만들어 공유한다. 도구 데이터는 기본 비공개(→ 4장).

### Phase B — 공유 루프 강화
- 도구→카드 원탭 공유의 마찰 최소화, 카드에서 사전 항목 자동 링크(지뢰 플래그).
- 카테고리·지역별 카드 모아보기 (분포는 아직 아님 — 개별 카드의 나열).
- 견적 디코더: 견적서 캡처 → 구조화 → 카드로 공유. 고신뢰(A등급) 데이터 수집기이자 바이럴 포인트.

### Phase C — 분포 공개 (임계 잠금해제)
카테고리×지역 단위로 공유 카드가 임계(n≥10)를 넘는 곳부터 실거래 분포를 연다.
"○○명이 실제로 낸 금액 기준 · 중앙값 ○○만원" — 이 문장을 쓸 수 없으면 그 화면은 만들지 않는다.
분포가 열리면 개인의 협상 기준선이 생긴다: *"이 카테고리 중앙값이 X인데 견적이 X+40%네요."*

**단계 승격 규칙:** 다음 Phase의 기능은 이전 Phase가 도그푸딩 검증을 통과하기 전에 만들지 않는다.

## 4. 공유는 명시적 행위다 — 도구 데이터는 새지 않는다

- 트래커의 expense는 **기본 비공개, 사용자 소유**다. 도구를 쓰기 위해 공유를 강요받지 않는다.
- 공유는 **스냅샷 복사**다: 지출을 카드로 공유하는 순간의 값이 카드에 복사된다. 이후 트래커에서 수정해도 카드는 사용자가 직접 고치기 전까지 그대로다 — 사적 데이터와 공적 데이터의 물리적 분리(스키마 수준).
- 공유 시 카드에 무엇이 담기는지 그 화면에서 전부 보여준다. 포스트·카드는 언제든 삭제 가능하고, 삭제하면 집계에서도 빠진다.
- 시기는 월 단위, 지역은 광역 단위로만 카드에 담는다 — 특정 가능성을 스키마에서 차단한다.

## 5. 데이터 철학 (이 제품의 심장)

### 5.1 가격은 "정가"가 아니라 "실거래 분포"다
업체 공시가는 존재하지 않거나 거짓이다. 단일값(min/max) ❌ → **표본 기반**(중앙값·사분위·표본수·신뢰 등급) ✅

### 5.2 데이터의 원천 우선순위 (v3)
1. **공유 카드 (주력).** 사용자가 *자기 이야기를 하려고* 공유한 실지출 — 커뮤니티 맥락(글·댓글)이 붙어 있어 검증 가능성이 높다.
2. **견적 디코더 (고신뢰, Phase B).** 원본 증빙이 있는 인증 카드.
3. **커뮤니티 후기 마이닝 (보강).** 외부 공개 후기에서 *사실만* 추출. 본문 저장·재공개 금지.

### 5.3 임계 규칙 — 얇은 표본은 침묵한다
- 카테고리×지역 표본 **n < 10: 분포 비공개** ("아직 표본이 부족해요. 공유하면 빨라져요"로 정직하게)
- n ≥ 10: 공개하되 표본수·기간·신뢰 등급 상시 표기
- 모든 분포 화면에 "이 숫자는 어떻게 만들어졌나" 링크 (산출 방법 공개)

### 5.4 신뢰 등급 — 모든 데이터 포인트는 출처 등급을 가진다
| 등급 | 원천 | 가중치 |
|------|------|--------|
| A (인증) | 견적 디코더 원본 증빙 | 최고 |
| B (공유 카드) | 트래커 연동 or 직접 작성 공유 카드 | 높음 |
| C (자가 보고) | 맥락 없는 단건 수기 입력 | 보통 |
| D (마이닝) | 외부 공개 후기 추출 | 참고 |
분포 계산은 등급 가중, 화면에는 등급 구성 표기.

### 5.5 오염 방어 — 커뮤니티는 업체의 공격 표면이다
자기 가격은 낮게, 경쟁사는 높게, 후기를 가장한 광고 — 전부 유인이 존재한다:
- 이상치 탐지 (카테고리별 분포 대비 자동 플래그, 사람 검토 큐)
- 계정당 카드 rate limit, 단기간 동일 업체 집중 언급 탐지, 신규 계정의 첫 포스트 검토
- A등급(증빙) 데이터가 앵커 — 공유 카드가 인증 분포와 크게 어긋나면 신뢰 하향
- 오염 시도가 확인된 업체는 내부 기록 (공개 비방은 하지 않는다 — 우리는 데이터로만 말한다)

### 5.6 비용 데이터 신뢰의 단일 급소: `scope`(포함 범위)
"스냅 100만원"이 촬영만인지 앨범·보정 포함인지 모르면 비교 불가다.
- **모든 지출·카드 레코드는 `scope`를 1급 필드로 반드시 받는다.** 숨은비용 사전이 곧 `scope` 체크리스트의 원천.
- 항목별 가격 변수(스냅의 시간·렌즈, 드레스의 벌수·브랜드급 등)는 `attributes JSONB`로 흡수한다.

### 5.7 업체는 선등록하지 않는다 — 데이터에서 자라난다
크롤링 선등록 ❌. 사용자가 업체명을 적으면 그 순간 `vendor` upsert. 등록된 업체는 전부 실데이터 1건 이상.

### 5.8 크롤링은 회색지대다 — 선을 지킨다
- ✅ robots.txt·약관 존중, 공개 페이지만, rate limit 준수
- ✅ 본문 복제·전시 ❌ → 사실 데이터만 추출
- 🚫 폐쇄형 카페 로그인/접근통제 우회 코드는 절대 만들지 않는다
- 🚫 작성자 개인정보(연락처 등) 절대 수집 금지

### 5.9 익명화는 k-익명성 기준
공유 카드는 본인이 공개를 선택한 것이지만, **집계·분포는 k-익명성을 지킨다**: 셀 단위 k≥5 미만이면
상위 버킷으로 병합. 카드 자체도 시기=월, 지역=광역 단위 이상으로만 담는다(4장). 닉네임 기반이며
실명·연락처를 요구하지 않는다.

## 6. 기술 스택과 데이터 모델

- **PostgreSQL 단일** (MVP). Neo4j는 보류 — 관계 질의가 실제 병목이 되기 전까지 도입하지 않는다.
- **FastAPI + Next.js.** 백엔드는 얇게, 데이터 모델에 투자한다.
- OCR+LLM 파이프라인은 Phase B 전까지 만들지 않는다 (승격 규칙).

핵심 테이블 (개념):

```
user            닉네임 기반 계정. 실명·연락처 없음
post            준비 기록 — 글 본문, 작성자
post_card       포스트에 첨부된 지출 카드 — 공유 시점의 스냅샷
                (category, amount, scope[], attributes, vendor_name, region, paid_month)
comment         댓글
taxonomy        카테고리 트리 (스드메/예식장/스냅/한복/... versioned)
expense         예산 트래커 항목 — 사용자 소유, 기본 비공개. 카드의 원재료
vendor          upsert-only. 실데이터에서만 생성
fee_glossary    숨은비용 사전 — scope 체크리스트의 원천, SEO 유입구
quote           (Phase B) 견적 디코더 결과
```

원칙: `expense`(사적 도구)와 `post_card`(공적 기록)를 물리적으로 분리한다. 도구 데이터가 사용자의
명시적 공유 행위 없이 공개되는 경로가 스키마 수준에서 존재하면 안 된다.

## 7. 작업 방식 (반드시 상시 적용)

### 7.1 도그푸딩 — 1호 사용자이자 1호 기여자는 우리 자신이다
- 메인테이너 본인의 결혼 준비(2027년 3월, 수도권)가 첫 콘텐츠다. 플래너 미팅·견적·실지출·짜증 포인트를 실제로 기록하고 **실제로 공유한다.**
- 가짜 데이터·가짜 계정으로 피드를 채우지 않는다. 빈 피드는 초대로 풀지, 연출로 풀지 않는다.
- 새 기능마다 묻는다: **"내가 지금 내 결혼 준비에 이걸 실제로 쓸까? 이 포스트를 실제로 올릴까?"** 안 쓸 기능은 만들지 않는다.

### 7.2 페르소나 테스트 — 모든 기능은 페르소나 세션으로 검증한다
기능을 "완성"이라 부르기 전에 `PERSONAS.md`의 각 페르소나로 첫 세션을 끝까지 걸어본다. 특히 **적대적 페르소나(오염 시도 업체, 프라이버시 민감 사용자)를 통과 못 하면 완성이 아니다.**

### 7.3 신뢰는 카피에서도 지킨다
과장 금지. "최저가 보장" 같은 말은 우리 어휘에 없다. 우리가 아는 것과 모르는 것을 구분해서 말한다. 표본이 부족하면 부족하다고 쓴다.
